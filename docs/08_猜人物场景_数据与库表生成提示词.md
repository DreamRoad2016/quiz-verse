## 0. 角色设定（给大模型的指令）

你是一名资深后端架构师，熟悉 **PostgreSQL**（含 `jsonb`、约束、索引、`text[]`、`GIN` 等）与游戏化产品设计。你的任务是：在充分理解下方「产品是什么、硬性规则是什么」的前提下，为 **指定的猜人物场景** 输出 **合理、可实现、可演进** 的数据库设计（表、字段、索引、必要的枚举或校验策略），并简要说明 **每列如何参与服务端比对**（绿 / 黄 / 灰、数值 ↑↓ 等）。

输出请使用 **中文**，并包含：

1. **设计摘要**（3～8 句）：场景边界、与「作品 / 大类」的关系。  
2. **ER 或表清单**（列表即可）。  
3. **DDL**：`CREATE TABLE` / `CREATE INDEX` / 必要 `COMMENT ON`；符合 PostgreSQL 习惯。  
4. **`work_column`（列元数据）示例数据**：至少列 key、展示名、数据类型、比对规则类型、排序。  
5. **样例行**：1～3 条 `INSERT` 或 JSON 示例，展示 `attrs` 形态。  
6. **风险与扩展**：例如多值字段、nullable、剧透字段、索引与导入流水线。

不要编写具体业务代码（Java / TS）；除非自定义区要求，否则不要臆造未公开的剧情细节，**不确定的字段可标注为可选或待运营填写**。

## 1. 产品是什么（必须理解）

我们要做的是一款 **类 Wordle / LoLdle** 的 **「猜人物」** 游戏（内部也称 Guess All / 竞猜宇宙）：

- 每一局系统**秘密选定**题库中的一个 **人物（角色）** 作为答案。  
- 玩家在有限次数内，每次从题库中 **选一个角色提交**。  
- 服务端将 **猜测角色** 与 **答案角色** 按 **预先声明的若干列（属性维度）** 逐列比对，返回每列的 **着色反馈**（例如 **绿 = 完全匹配、黄 = 部分相关、灰 = 不匹配**；数值类还可 **↑ / ↓** 提示相对高低）。  
- 猜中答案或次数用尽则本局结束。

**与标杆网页游戏 guessassin.xyz 的关键差异（必须遵守）**：

- 标杆把整库拉到浏览器、在客户端比对；**我们不采用**。  
- **所有比对在服务端完成**；客户端/浏览器 **不得** 拿到完整 `attrs` 或答案行用于推导。  
- 开局时服务端只下发 **体量可控的「联想 brief」**：每条人物仅 **`id` + 规范显示名 + 别称（及可选拼音）**，用于搜索与展示；**完整属性留在服务端**（见立项 ADR-002）。

当前阶段以 **自建网页 + Spring Boot API** 验证玩法为主，**不要求**小程序形态。

## 2. 主要逻辑与需求（硬性规则）

### 2.1 对局与题库边界

- **影视剧 / 综艺 / 单 IP 虚构作品** 场景：**一局只对应一部作品（一个 work）**，题库仅为该作品内的角色。  
  - **原因**：不同作品的列语义无法强行统一（例如《甄嬛传》的位分、居所与《三体》的职务、组织不在同一套维度）；混池会导致大量「不适用」或列名失真。  
- **历史人物等跨时代共池** 场景（未来）：可另设大类，使用 **可跨人物类比的列**（如朝代、性别、领域等）；与「单作品锁局」是 **不同大类或不同 `pool_type`**，不必强行共用同一套 `work_column`。

### 2.2 数据建模原则（已与人类架构师对齐的方向）

- 使用 **PostgreSQL**。  
- **作品维度 + 该剧 schema**：每个作品（`work`）有独立的 **列定义（元数据）**，角色行的可变属性优先放在 **`jsonb`（如 `attrs`）** 中，避免「每剧一张完全不同的宽表」导致 ORM 与迁移爆炸。  
- 仍可保留少量 **跨剧通用、可索引** 的关系列（例如 `work_id`、规范名、`sort_order`、`is_active` 等），便于列表与运营；**是否抽通用列**由场景决定，不强制填满。  
- **列元数据表**（名称可叫 `work_column` / `scenario_column` 等）描述：列 key、`jsonb` 内路径、展示名、数据类型、比对规则类型、显示顺序、是否参与本模式等。服务端比对引擎 **按元数据驱动**，而非把列名写死在代码里。

### 2.3 比对与隐私

- 比对输入：答案角色 id、猜测角色 id（均在服务端解析）。  
- 比对输出：每列着色结果 + 必要箭头；**不在中途泄露答案身份**。  
- brief 与 attrs 分离：**attrs 永不下发给客户端**。

### 2.4 联想与展示

- 联想列表仅包含库中维护的 **规范名与别称**（别称可为数组或子表）；是否需要 **拼音** 由实现决定，可在列元数据或全局配置中说明。

### 2.5 比对类型（需与实现一致时可引用）

列的比对在服务端按类型计算，常见类型包括（命名可与 `04_单局猜测算法.md` 对齐）：

- **精确匹配**（枚举单值、布尔等）。  
- **集合 / 多值**：交集非空为黄、完全相等为绿等（规则需在元数据或文档中写清）。  
- **文本**：完全相等 / 规范化后相等 / 子串等（按场景选择）。  
- **数值**：相等为绿，不等为灰并带 ↑↓（若业务需要）。  
- **不适用**：若某列对该角色无值，需定义是灰、隐藏还是「不适用」单独样式（请在设计中明确）。

## 4. 数据库设计应交付的内容清单

大模型生成结果时，请尽量覆盖：

| 项 | 说明 |
| --- | --- |
| `work`（或等价） | 作品实体：id、标题、类型、可选年代、slug、启用状态 |
| `work_character`（或等价） | 角色行：`work_id`、主键、`display_name`、`aliases`（`text[]` 或 json）、`attrs jsonb`、审计字段 |
| `work_column` | 列元数据：key、label、value_type、compare_rule、`sort_order`、是否必填等 |
| 索引 | 至少：`work_id`；`display_name` / `aliases` 检索策略（如 `GIN (gin_trgm_ops)` 需声明扩展）；`attrs` 若需可按 `jsonb_path_ops` 说明慎用场景 |
| 约束 | `work_id` 外键、`attrs` 与 `work_column` 的 key 集合一致性（应用层或 `CHECK` 择一说明） |
| 版本与迁移 | 是否需 `schema_version` 列便于改列 |

## 5. 生成时的默认假设（若自定义区未写明）

- 单作品 MVP：**角色数量级 ≤ 500**（可调整）。  
- 单次开局下发 **全作品 brief 列表**（与 LoL Demo 一致量级假设）。  
- UTF-8，中文显示名合法。  
- 不在 DDL 中硬编码版权图片 URL；媒体字段若有，标注为可选。

---

## 6. 反模式（请避免）

- 把多部影视剧的角色塞进同一表且 **无 `work_id` 强约束**，却共用一套「甄嬛传列」。  
- 把完整 `attrs` 或答案下发给前端。  
- 仅有一张宽表、无任何列元数据，导致每加一列都要改代码与迁移。  
- 过度依赖无索引的 `jsonb` 全表扫描做线上热路径（应说明冷热路径）。
