## ADR-001：题库组织采用"**大类约束 + 小标签并集**"

- **日期**：2026-04-29
- **状态**：✅ 采用
- **决策人**：项目发起人

### 背景

题材跨度极大（古代皇帝、选秀成员、足球明星、科技大佬…），不同人物之间几乎找不到稳定的公共属性。例如皇帝有"朝代/谥号/在位年数"，选秀成员有"出道公司/团体/选秀节目"，足球明星有"位置/俱乐部/国家队"——硬塞进一张通用表会让 80% 的字段都是 null。

### 决策

把题库分为**大类（Category）**和**小标签（Tag）**两层：

1. **大类**：每个大类拥有**一套固定的属性 schema** 与**一套固定的比对模板**。例如：
   - `history`（历史人物）
   - `arts`（文人/艺术家）
   - `entertainment`（演艺/明星）
   - `sports`（体育）
   - `sci_tech`（科技/学者）
   - `business`（商业人物）
   - `grassroots`（网红/草根）
2. **小标签**：依附于某个大类，是题材的细分维度。例如 `history` 下有 `三国`、`秦汉`、`帝王`、`武将`、`诗人`；`entertainment` 下有 `选秀成员`、`流量小生`、`实力派`、`偶像团体`；`sports` 下有 `足球`、`篮球`、`电竞`。
3. **进入对局**时玩家必须先选 **1 个大类**，再在该大类内选 **0~N 个小标签**：
   - 若不选标签 → 全大类为题库
   - 若选了 N 个标签 → 题库 = **任一标签命中即入选（并集）**
4. **不允许跨大类混合**——即使界面上做"我全都要"也按多个独立局处理。

### 收益

| 维度 | 收益 |
| --- | --- |
| 数据建模 | 共性字段进 `person` 主表，类别专属字段进 `person_<category>_attrs` 子表，强类型可索引 |
| 比对算法 | 每大类一份比对模板（列固定、类型固定），UI 表格列固定，玩家体验一致 |
| 内容运营 | 录入时只需在该大类的 schema 内填字段，不必为每个新人物决定要不要展示某列 |
| 扩展 | 新增大类只需建一张子表 + 一份模板，不影响已有大类 |
| 前端 | 每大类可定制专属表格列、提示语、配色，"古代皇帝局"和"NBA 局"长得不一样但都自洽 |

### 替代方案

- ❌ **统一通用 schema + jsonb attrs**：跨题材时模板变"通用模板"，列退化为只剩名字/性别/出生年，可玩性大幅降低。
- ❌ **完全独立的子产品（每题材独立小程序）**：用户分流，社交分享被打散，运营成本翻倍。

### 影响

- `03_数据模型设计.md` 重写为"主表 + 大类专属子表"。
- `04_单局猜测算法.md` 比对模板按大类硬编码。
- 前端选标签界面需要二级结构：先选大类，再多选标签。

## ADR-002：所有比对走**服务端**，不下发完整题库

- **日期**：2026-04-29
- **状态**：✅ 采用
- **决策人**：项目发起人

### 背景

标杆产品 `guessassin.xyz` 是把整个题库一次性下发给前端，比对在 JS 中完成。这个做法对**单人玩家**而言响应最快、成本最低，但带来 4 个问题：

1. 题库**版权与积累**直接暴露给客户端（可被一次性"扒库"）。
2. 内存里看得到 `answer`，技术玩家可作弊（单人无所谓，对战必死）。
3. 题库扩到几千条以后，单次下发会变臃肿。
4. 单人/对战两套逻辑导致代码分叉，维护成本翻倍。

### 决策

**所有模式（单人 + 对战）的比对都在服务端做**，客户端只发"`POST /matches/:id/guess` { guessPersonId }"，服务端返回着色后的逐列结果。

- 客户端只缓存"用于联想搜索的人物 brief 列表"（id/姓名/别名/拼音），不含 attrs，不能用于反推答案。
- 题库不下发，整库存储在服务端。
- 单条猜测端到端延迟目标 < 200ms（同区），玩家不可感知。

### 收益

- 单人/对战**逻辑统一**，比对引擎只一份代码。
- 反作弊的天花板由服务端把控；玩家想作弊必须"猜遍题库"——成本自然抑制。
- 题库可任意扩张（万条以上不影响客户端体积）。
- 题库内容可以增量更新而不需要发版。

### 替代方案

- ❌ **单人客户端比对、对战服务端比对**：两套代码，单人模式题库泄漏，且未来想升级单人模式排行榜也要重做。
- ❌ **全量预下发但加密**：加密对抗只是延缓而非阻止，且增加首次加载延迟，不值得。

### 影响

- `04_单局猜测算法.md` 全面改为服务端比对的协议。
- `06_技术栈与架构.md` 后端比对引擎是核心服务。

## ADR-003：**必须独立部署后端服务**，"纯小程序云开发"不能承担长期形态

- **日期**：2026-04-29
- **状态**：✅ 采用（MVP 期可临时使用云开发的"云函数 + 云数据库"过渡）
- **决策人**：项目发起人

### 背景

项目主问："要做对战的话，是不是必须搭个独立云服务，小程序自己搞不定？"——是的。需要把"为什么"和"边界在哪"写清楚。

### 微信小程序的限制

| 维度 | 限制 |
| --- | --- |
| 运行环境 | 沙箱式 JS，**不能开端口、不能监听连接** |
| 长连接 | 仅能作为**客户端**通过 `wx.connectSocket` 连出去（连别人的服务端），不能自己当服务端 |
| 后台运行 | 切后台或锁屏可能被冻结，无法持续承担房间状态 |
| 跨用户共享状态 | 小程序之间不能直接通信，必须通过共同的服务端中转 |
| 数据库 | 本地 `Storage` 仅限本机；想多人共享必须中心化 |
| 业务安全 | 客户端代码可被反编译，不能存放敏感逻辑 |

**对战的本质需求**：多名玩家共享同一份"对局状态"（答案、谁猜了什么、谁先命中），这必须有"第三方裁判"——而沙箱化的小程序不具备做裁判的条件。

### 决策

**必须有独立可控的后端服务。** 三种部署形态选其一：

#### 方案 A：阿里云 / 腾讯云自建（推荐长期形态）

- 云服务器（ECS / CVM）跑 Node.js 或 Go 后端
- 自建 Postgres + Redis（或买云数据库）
- 域名 + HTTPS + 备案
- WebSocket 直连服务端
- **优点**：完全自主、易接 CI/CD、扩容自由、迁云容易
- **缺点**：需要自己管运维（备案、证书、监控、安全组、扩容）
- **MVP 月成本估**：1 核 2G ECS（30~60 元）+ Postgres（30~80 元）+ Redis（10~30 元）+ 流量 ≈ **80~200 元/月**

#### 方案 B：微信云开发（云函数 + 云数据库 + 实时数据库）（MVP 推荐）

- 直接用微信云开发的"云函数"做后端比对
- "云数据库"存题库
- "实时数据库"做房间状态同步（订阅 / 监听变更）
- **优点**：免备案、免域名、免证书、免运维；登录态天然
- **缺点**：长期看有平台锁定，复杂业务逻辑写在云函数里维护成本高；高并发下成本未必便宜；想做 H5/App 镜像时全部白搭
- **MVP 月成本估**：免费额度内基本不花钱

#### 方案 C：Serverless（云函数 + RDS）

- 阿里函数计算 / 腾讯云函数 + 云数据库
- WebSocket 用专门的"实时通信" PaaS（如腾讯云 IM / 阿里 MNS）
- **优点**：弹性、按量计费
- **缺点**：WS 长连接在 Serverless 上麻烦，需要专门 PaaS

### 最终路线建议

```
MVP（V0~V1）：方案 B 微信云开发  ──►  公测/对战正式上线（V1.0+）：方案 A 自建云服务
```

- **MVP 阶段不立马自建**：用云开发的"云函数 + 云数据库"快速跑出 demo，零成本验证玩法。
- **对战正式版迁移到方案 A**：因为 ① 实时连接质量、② 业务逻辑可控、③ 跨端镜像（H5/App）的需要，迁出云开发是必经路径。
- 数据模型从一开始就按"数据库中立"设计（不依赖云开发特有 SDK），便于将来迁移。

### 边界明确

| 场景 | 必须有后端？ |
| --- | --- |
| 单人本地小练（仅看说明、不联网） | 不必须，但很受限 |
| 单人在线（拉题库、记战绩） | 必须 |
| 对战房间 | **必须，且必须是"权威裁判"**，不能让客户端自己判胜负 |
| 排行榜 / 段位 | 必须 |
| UGC 题库 | 必须 |

### 影响

- `06_技术栈与架构.md` 增加"为什么必须独立后端 + 云方案对比"章节。
- `07_里程碑与风险.md` 把"MVP 用云开发"放进 M1 实施步骤里。

## ADR-004：**同一人物可归属多个大类**（多分类）

- **日期**：2026-04-29
- **状态**：✅ 采用
- **决策人**：项目发起人

### 背景

很多真实人物跨界明显——姚明（篮球 + 演艺 + 商业）、周杰伦（演艺 + 电竞投资人 + 商业）、苏轼（文学 + 历史官员）、张朝阳（科技 + 物理学者）。如果强制单大类归属，要么出现"姚明只能在体育局猜不到"，要么需要把同一人物拆成多条数据，运营成本反而更高。

### 决策

人物的大类归属**通过"是否在该大类的专属子表里有记录"自然表达**：

- 主表 `person` 不再带 `category_id` 单值字段
- 同一 `person.id` 可以在 `person_sports_attrs`、`person_entertainment_attrs` 等多张子表中**同时存在记录**
- 每张子表内的字段是该人物**在该大类视角下**的画像（姚明的 sports 记录是中锋/休斯顿火箭；entertainment 记录是演员/CBA 主席/参演节目）
- 选某大类开局时，题库就是"该大类子表里有记录的人"
- 比对时按当前对局的大类读对应子表，不会串味

### 收益

- **同一人物能在不同题库中作为合法答案**，玩法体验更丰富
- **多分类不需要额外建模**——子表已经天然分离
- **录入工作可分阶段**：MVP 期先把姚明录到 sports，后期补 entertainment 视图，互不影响
- **大类列表稳定**——大类数 = 子表数 = 比对模板数，运营动态加大类的需求可以延后到无限远（ADR 决策保留）

### 替代方案

- ❌ **person 加 `categories[]` 多值数组**：仍需要每张子表里手动校验"该 person 是否被允许写入"，校验冗余且容易漏。
- ❌ **每个大类拆成独立 person 行**：姚明会变两条记录，命中判定要做 alias 链，复杂且违反直觉。

### 影响

- `03_数据模型设计.md` 中 `person` 表去掉 `category_id`；新增"通过子表存在性表达多分类"小节。
- 题库筛选 SQL 改为按"JOIN 当前大类子表"过滤。
- `tag` 表仍 `category_id` 必填（标签依附大类），但 `person_tag` 关系的合法性需校验"该 person 在该大类子表存在"。
- 录入工作流：每条人物可勾选多个大类，对应多个 `attrs` 表单（哪个大类填了哪份就属于哪个大类）。

## ADR-005：MVP 后端栈选 **Java（Spring Boot）+ PostgreSQL + Redis**，**本地优先**开发

- **日期**：2026-04-29（首版）/ 2026-04-29 修订
- **状态**：⚠️ **部分被 ADR-007 / ADR-008 修订**（栈仍为 Java + Spring Boot；**内容层 MVP 不用 PG**，对局可先内存）
- **决策人**：项目发起人
- **修订说明**：首版选 MySQL（基于"团队最熟"），权衡后改为 **PostgreSQL 15**——多值集合（朝代/势力/历史战队/擅长英雄等）和中文联想搜索是本项目的核心数据场景，PG 在这两点上原生强于 MySQL，且对 Java 生态完全无门槛。2026-07 重写后：比对与题库先走题包文件（ADR-007），PG 推迟到需要账号/战绩/多人录入时再引入。

### 背景

项目发起人最熟悉的栈是 **Java**，且也用过 Python / PHP，并不绑定 MySQL。希望先在本地完整跑通整套（后端 + 数据库 + 缓存 + Web 测试页），稳定后再迁云。

### 决策

| 层 | 选型 | 备注 |
| --- | --- | --- |
| 后端运行时 | **Java 17 + Spring Boot 3** | 团队最熟，生态成熟 |
| ORM | MyBatis-Plus 或 Spring Data JPA | 任选其一；对 PG 完全友好 |
| 数据库 | **PostgreSQL 15** | 多值集合 `text[]` + GIN 索引、`jsonb`、`pg_trgm` 中文模糊匹配原生支持 |
| 缓存 | **Redis 7** | 对局状态 + 题库 brief 缓存 |
| 前端验证 | **本地 HTML + JS**（不用 Taro/小程序） | 先跑通逻辑，后改造为小程序 |
| 部署 | **本地** Docker 起 PostgreSQL + Redis；Java 直接 IDEA 跑 | 验证完再上云 |

### 为什么不选 MySQL（关键对比）

| 场景 | MySQL 8 | PostgreSQL 15 |
| --- | --- | --- |
| 多值集合（核心比对算法依赖） | JSON 列 + `JSON_OVERLAPS`，建索引需虚拟列 workaround | **`text[]` + GIN 索引原生支持**，`@>` `&&` 操作符直接用 |
| 中文联想搜索 | `LIKE '%X%'` 全表扫；ngram 全文索引中文一般；常需外挂 ES | **`pg_trgm` 一句 SQL 解决模糊匹配**；`zhparser` 做分词 |
| `jsonb` 性能 | JSON 是压缩文本，查询要解析 | jsonb 是二进制，查询不解析 |
| 复杂查询（CTE / 窗口） | 8.0 才完整支持 | 原生强 |
| DDL 安全性 | 部分阻塞写 | 全在事务内可回滚 |
| 学习成本（对 Java 用户） | 0 | 半天～1 天 |
| 国内 RDS 月成本 | 30~50 元 | 50~80 元（差距可忽略） |

**关键判断**：本项目的核心查询是"多值集合的相交比对"和"中文人名联想"，**PG 在这两个场景是降维优势**。学习成本对 Java 用户极低（JDBC、JPA、MyBatis 全部对 PG 友好）。月成本差距对个人项目可忽略。

### 开发路线（本地 → 云端）

```
M0~M2 全部在本地：
  本地 IDEA Java + 本地 PostgreSQL + 本地 Redis + 本地 HTML/JS 浏览器测试
        ↓ 玩法 + 数据 + 接口跑通
M3 单人功能稳定后再上云：
  阿里云/腾讯云 ECS + RDS for PostgreSQL + Redis 云版
        ↓ 备案 + HTTPS（这一步异步启动）
M4 改造小程序前端：
  HTML 改造为微信小程序页面（接口契约不变）
```

### 收益

- 心智负担小，纯 Java + Spring Boot + PostgreSQL（PG 对 Java 用户半天上手）
- **本地全栈打磨完，再迁云**，云上调试时间最小化
- HTML 测试页 = 后期小程序的"接口验证用例库"，**接口契约稳定后小程序只是换皮**
- 备案/HTTPS 等"非核心却阻塞"的步骤推到后期并行处理

### 影响 / 注意事项

- **多值集合用 `text[]`**（不是 JSON），便于 GIN 索引和 `&&` `@>` 操作符直接用
- **联想搜索用 `pg_trgm`**（GIN 索引，支持 `LIKE '%中文%'` 高效模糊匹配），上千条数据无压力，几万条也能扛
- **复合属性用 `jsonb`**（如未来扩展的 `extra` 字段）
- 应用层把"接口契约"做厚（DTO、参数校验），减少未来改前端的工作量
- 上云时选阿里云 RDS for PostgreSQL 或腾讯云 TencentDB for PostgreSQL（都是稳定提供），月成本 50~80 元起

## ADR-006：对战采用 **REST 短轮询 + Redis 状态**，**MVP 期不引入 WebSocket**

- **日期**：2026-04-29
- **状态**：✅ 采用（V2 视体验情况再升级到 WS）
- **决策人**：项目发起人

### 背景

之前 `05_对战模式设计.md` 默认上 WebSocket Hub，做完整的实时事件推送。这套实现复杂（断线重连、心跳、订阅、广播队列…），对 Java 后端、本地 → 云迁移、小程序后台/锁屏的兼容都有挑战。

发起人提议：**一局对战在 Redis 里存一个 UUID + 状态对象，玩家每次"猜一下"时直接读写 Redis 比对，无需长连接**。客户端定时拉取对局快照即可看到对手进度。

### 决策

对战采用以下结构（详见 `05_对战模式设计.md` 重写版）：

#### 1. 对局状态存 Redis（活跃期），TTL 自动过期

```
match:{uuid}                  Hash   对局元数据（answerId、category、tagIds、mode、maxGuesses、winner、status、startedAt、endedAt）
match:{uuid}:players          Set    参与玩家 userId 列表
match:{uuid}:player:{userId}  Hash   单玩家进度（guessCount、hit、durationMs）
match:{uuid}:player:{userId}:guesses  List  该玩家已提交的猜测明细（含比对结果）
match:{uuid}:lock             SetNX  原子锁，用于首杀写入

TTL 全部 30~120 分钟（按对战模式），到期自动清理 Redis
```

#### 2. 接口走 REST，无 WebSocket

| 接口 | 作用 |
| --- | --- |
| `POST /matches/1v1` 或 `/matches/multi` | 创建对局，返回 uuid |
| `POST /matches/{uuid}/join` | 加入对局 |
| `POST /matches/{uuid}/start` | 房主开始（多人模式需要） |
| `POST /matches/{uuid}/guess` | 提交一次猜测，**服务端比对 + 首杀判定** |
| `GET  /matches/{uuid}` | 拉取对局快照（包含全员进度摘要） |
| `POST /matches/{uuid}/leave` | 离开 |

客户端**定时轮询** `GET /matches/{uuid}`（活跃时 2~3s，空闲时 8s，后台暂停轮询）。

#### 3. 1v1 与多人分两种模式

| 模式 | 触发 | 房主 | 胜负判定 |
| --- | --- | --- | --- |
| `1v1` | 第二人 join 即开始 | 无 | 谁先 hit / 谁猜测次数少 |
| `multi`（3~8 人） | 房主点击 start | 有 | first_hit / least_guesses / least_time 三选一 |

#### 4. 完成才入库

- 对战进行中**只用 Redis**（避免每次猜测写 PostgreSQL）
- 终局触发后**异步**写 `match` + `guess` 表用于战绩、回放、分析
- Redis 数据可保留到 TTL 自然过期（用于补偿写入失败时重试）

### 收益

- **不需要长连接基础设施**——后端就是普通 Spring Boot REST，部署/迁移成本最低
- **小程序友好**：小程序后台/锁屏会冻结 WS 但不影响 REST 短轮询；前台恢复继续轮即可
- **故障域清晰**：Redis 挂了对局丢，但单局短（< 5min）影响小；PostgreSQL 挂了不影响活跃对局
- **后续可平滑升级到 WS**：当对战对实时性要求变高，把"猜测提交"和"快照变更"事件接入 WS 即可，REST 接口保留作为兜底

### 折衷 / 体验差距

- "对手刚命中"的瞬间感知有 ≤ 3s 延迟（玩家可接受，且双方屏幕都看得到"XX 已猜中"）
- 服务端额外 QPS：`100 房间 × 4 人 × 0.5 QPS（2s 轮询）= 200 QPS`，对单机 Spring Boot 毫无压力
- 客户端轮询消耗少量电量/流量（自适应频率可压到很低）

### 替代方案

- ❌ **完整 WebSocket 实时推送**：体验最好但 MVP 工作量翻倍，故障域复杂，**留待 V2**。
- ❌ **微信小程序 IM 能力 / 实时数据库**：可绕过自建 WS，但被云开发锁定（违反 ADR-003）。
- ❌ **Server-Sent Events (SSE)**：小程序 `wx.connectSocket` 支持有限，且单向推送，不如 WS 通用，**不推荐**。

### 影响

- `05_对战模式设计.md` **大改**：WS 协议替换为 REST + 轮询，新增 Redis 数据结构章节、1v1 与 multi 区分。
- `06_技术栈与架构.md` 总体架构图去掉 WS Hub，改为 Redis-centric。
- `07_里程碑与风险.md` 把"WS 弱网体验"风险替换为"轮询延迟 / 电量"风险（更轻）。

## ADR-007：MVP 内容真相源采用 **题包文件（pack）**，不先上 PostgreSQL / 录入后台

- **日期**：2026-07-19
- **状态**：✅ 采用
- **决策人**：项目发起人

### 背景

早期实现把影视剧内容拆成「PG 表 + `fields.json` + enums + studio/admin」，字段定义散落多处，录入与维护成本高，且在玩法未充分验证时过早建设了运营平台。LoL Demo 的「一份扁平 JSON → 导入」路径反而更轻。

### 决策

1. 每个可玩题材是一个 **pack 目录**：
   - `pack.yaml`：id、标题、最大猜测次数、demo/official
   - `schema.yaml`：比对列（`identity` / `exact` / `set` / `number`）与枚举
   - `entities.json`：人物列表（`id` / `name` / `aliases` / `attrs`）
2. 服务端启动加载题包到内存；比对引擎按 schema 驱动。
3. 客户端只拉 brief；attrs 永不下发（延续 ADR-002）。
4. **PostgreSQL、通用 admin、录入台推迟**到：需要账号/战绩、多人协作录入、或题量与协作方式撑不住 Git 时再引入。届时可把 pack 导入库，或反过来以库导出 pack。

### 收益

- 内容可 Git diff；LLM 直接生成 JSON；改 schema 与数据同目录
- 玩法迭代不被库表迁移拖累
- 与 ADR-001「每局锁一大类」兼容：一个 pack ≈ 一个大类/作品池

### 替代方案

- ❌ 继续 PG + studio：验证期过重（已证伪）
- ❌ 纯前端 JSON 比对：违反 ADR-002

### 影响

- `03` / `06` / `07` / 甄嬛传与录入手册改为题包优先
- 代码包名 `net.quizverse`，资源目录 `src/main/resources/packs/`

## ADR-008：运行时定为 **Java 17 + Spring Boot 3**；对局存储默认内存、Redis 可选

- **日期**：2026-07-19
- **状态**：✅ 采用
- **决策人**：项目发起人

### 背景

历史代码停留在 Java 8 + Boot 2.6；规划文档已写 Java 17。干净重写时统一到 LTS 与现行 Boot 3（`jakarta.*`）。新电脑未必装 Redis，单人验证不应被基础设施挡住。

### 决策

| 项 | 选型 |
| --- | --- |
| 语言 / 框架 | Java 17 + Spring Boot 3.3 |
| 对局状态 | `quiz.match.store=memory`（默认）或 `redis` |
| 前端验证 | 静态 HTML（`index.html` / `guess.html`） |

### 收益

- 与生态主流一致；Boot 2 已停维护
- 零依赖即可 `mvn spring-boot:run` 玩通单人局

### 影响

- README / `06` 以本 ADR 为准；ADR-005 中「MVP 必装 PG」不再作为当前门槛
