# 3. 数据模型设计

> 本节遵循：
> - [ADR-001](./DECISIONS.md#adr-001题库组织采用大类约束--小标签并集)：每局**只锁定 1 个大类**，大类内多选标签取**并集**。
> - [ADR-004](./DECISIONS.md#adr-004同一人物可归属多个大类多分类)：**同一人物可归多个大类**，通过"在该大类的专属子表里有记录"自然表达归属。
> - [ADR-005](./DECISIONS.md#adr-005mvp-后端栈选-javaspring-boot--postgresql--redis本地优先开发)：数据库选 **PostgreSQL 15**，多值集合用 **`text[]` + GIN 索引**。

## 3.1 设计原则

1. **共性字段进 `person` 主表**——任何人物都能填的（姓名/性别/出生年/出生国/简介…）。**不带 `category_id`**。
2. **类别专属字段进 `person_<category>_attrs` 子表**——每个大类一张表，主键 = `person_id`。
3. **同一 `person.id` 可在多张子表同时存在**：姚明可同时拥有 `person_sports_attrs` 与 `person_entertainment_attrs` 两条记录，分别表达不同视角。
4. **每个大类一份固定的"比对模板"**——表格列、字段、比对类型在 schema 层面写死。
5. **多值集合用 `text[]`**（PG 原生数组类型），配合 GIN 索引，`&&`（相交）/`@>`（包含）操作符直接用。
6. **标签依附大类**（`tag.category_id` 必填），跨大类禁止互通。

## 3.3 主表 `person`（所有大类共享）

```sql
CREATE TYPE gender_t AS ENUM ('male','female','other','unknown');
CREATE TYPE era_t    AS ENUM ('ancient','medieval','early_modern','modern','contemporary');
CREATE TYPE status_t AS ENUM ('draft','published','hidden');

CREATE TABLE person (
  id              UUID         PRIMARY KEY DEFAULT gen_random_uuid(),
  display_name    VARCHAR(64)  NOT NULL,
  aliases         TEXT[],                                  -- 别名/字号/英文/外号
  pinyin          VARCHAR(128),
  gender          gender_t     NOT NULL,
  era             era_t,
  birth_year      INT,
  death_year      INT,
  birth_country   VARCHAR(64),
  birth_region    VARCHAR(128),
  summary         TEXT,
  wiki_url        TEXT,
  avatar_url      TEXT,
  difficulty      SMALLINT,
  quality_score   INT          NOT NULL DEFAULT 0,
  status          status_t     NOT NULL DEFAULT 'draft',
  created_at      TIMESTAMPTZ  NOT NULL DEFAULT now(),
  updated_at      TIMESTAMPTZ  NOT NULL DEFAULT now()
);

-- 索引
CREATE INDEX idx_person_status      ON person (status);
CREATE INDEX idx_person_birth_year  ON person (birth_year);
-- 联想搜索：trigram 模糊匹配（中文姓名/别名/拼音都走这里）
CREATE EXTENSION IF NOT EXISTS pg_trgm;
CREATE INDEX idx_person_name_trgm   ON person USING gin (display_name gin_trgm_ops);
CREATE INDEX idx_person_pinyin_trgm ON person USING gin (pinyin       gin_trgm_ops);
CREATE INDEX idx_person_aliases_gin ON person USING gin (aliases);    -- text[] 的 GIN 索引
```

> 注意：**没有 `category_id` 字段**。一个人物属于哪些大类，由"它在哪些 `person_*_attrs` 表里有记录"决定。

## 3.4 大类与标签

```sql
CREATE TABLE category (
  id              VARCHAR(32) PRIMARY KEY,             -- 'history', 'entertainment', ...
  name_cn         VARCHAR(64) NOT NULL,
  name_en         VARCHAR(64),
  description     TEXT,
  schema_id       VARCHAR(32) NOT NULL,                -- 关联比对模板（代码常量也行）
  display_order   INT,
  enabled         BOOLEAN     NOT NULL DEFAULT TRUE
);

CREATE TABLE tag (
  id              VARCHAR(64) PRIMARY KEY,             -- 'three_kingdoms', 'idol_survival_show'
  category_id     VARCHAR(32) NOT NULL REFERENCES category(id),
  parent_tag      VARCHAR(64) REFERENCES tag(id),
  name_cn         VARCHAR(64) NOT NULL,
  name_en         VARCHAR(64),
  description     TEXT,
  count_cache     INT         NOT NULL DEFAULT 0,      -- 关联人物数（缓存，定时刷新）
  enabled         BOOLEAN     NOT NULL DEFAULT TRUE
);
CREATE INDEX idx_tag_category ON tag (category_id);

CREATE TABLE person_tag (
  person_id       UUID        NOT NULL REFERENCES person(id) ON DELETE CASCADE,
  tag_id          VARCHAR(64) NOT NULL REFERENCES tag(id),
  PRIMARY KEY (person_id, tag_id)
);
CREATE INDEX idx_person_tag_tag ON person_tag (tag_id);
```

> **业务约束（应用层校验）**：写入 `person_tag(person_id, tag_id)` 时，必须保证 `person` 在 `tag.category_id` 对应的子表中**有记录**。否则用户在该大类局里筛标签会出现"含此 tag 但 attrs 缺失"的脏数据。

## 3.5 大类专属属性子表

> 每张子表的主键 = `person_id`，1:1 关联人物**在该大类视角下**的画像。
> 同一 `person.id` 可在多个子表同时出现，互不干扰。

### 3.5.1 `person_history_attrs`（历史人物）

```sql
CREATE TABLE person_history_attrs (
  person_id       UUID PRIMARY KEY REFERENCES person(id) ON DELETE CASCADE,
  dynasty         TEXT[],           -- {'东汉','三国-蜀汉'}
  factions        TEXT[],           -- {'蜀'}
  roles           TEXT[],           -- {'皇帝','武将','谋士'}
  titles          TEXT[],           -- {'昭烈帝','汉中王'}
  active_regions  TEXT[],           -- {'巴蜀','荆州'}
  era_century     SMALLINT,         -- 主要活动世纪（数值，可↑↓）
  reign_start     INT,              -- 在位起始年（仅帝王，可空）
  reign_end       INT,
  notes           TEXT
);
-- 多值字段都建 GIN 索引，支持 &&（相交）/ @>（包含）高效查询
CREATE INDEX idx_hist_dynasty ON person_history_attrs USING gin (dynasty);
CREATE INDEX idx_hist_roles   ON person_history_attrs USING gin (roles);
```

### 3.5.2 `person_entertainment_attrs`（演艺/明星）

```sql
CREATE TABLE person_entertainment_attrs (
```


## 3.8 题库筛选规则（PostgreSQL）

入参：`{ categoryId, tagIds: [...], difficulty?: [1..5] }`

```sql
-- 不选标签：该大类全部
SELECT p.id, p.display_name
FROM person p
JOIN person_history_attrs h ON h.person_id = p.id   -- 按大类切换 JOIN 不同子表
WHERE p.status = 'published'
  AND (:difficulty IS NULL OR p.difficulty = ANY(:difficulty));

-- 选了 N 个标签：并集（任一标签命中即入选）
SELECT DISTINCT p.id, p.display_name
FROM person p
JOIN person_history_attrs h ON h.person_id = p.id
JOIN person_tag pt          ON pt.person_id = p.id
WHERE p.status = 'published'
  AND pt.tag_id = ANY(:tagIds);
```

> 关键：**永远先 JOIN 当前大类的 attrs 子表**，自动排除"不属于该大类"的人。
> 实现层用 `Map<categoryId, attrTableName>` 做 SQL 拼装，避免硬编码。

### 3.8.1 比对算法用到的多值集合操作

PG 数组操作符正好对应 [04_单局猜测算法](./04_单局猜测算法.md) 的 `set` 比对类型：

```sql
-- 是否相交（set 比对：partial 判断）
SELECT a.dynasty && b.dynasty                AS has_overlap;
-- 交集元素（set 比对：matched 列表）
SELECT array(SELECT unnest(a.dynasty) INTERSECT SELECT unnest(b.dynasty)) AS matched;
-- 是否完全相等（set 比对：exact 判断）
SELECT a.dynasty = b.dynasty                  AS is_equal;
```

> 实际比对建议**在 Java 应用层做**（详见 [04 §4.5](./04_单局猜测算法.md#45-比对引擎伪代码)）：从 PG 把两条记录的 `text[]` 直接读到 `Set<String>`，业务规则更清晰、可单测。SQL 操作符仅在管理后台批量分析时用。

## 3.9 联想 brief 查询

```sql
-- 给某大类下的题库人物提供联想（只返回 id+名+别名+拼音，不含 attrs）
SELECT p.id, p.display_name, p.aliases, p.pinyin
FROM person p
JOIN person_<category>_attrs a ON a.person_id = p.id
WHERE p.status = 'published'
  AND ( :tagIds IS NULL
        OR EXISTS (SELECT 1 FROM person_tag pt
                   WHERE pt.person_id = p.id AND pt.tag_id = ANY(:tagIds)) );
```

按 `(categoryId, tagSet)` 做 5min Redis 缓存。

### 3.9.1 联想搜索（用户输入即匹配）

得益于 `pg_trgm` + GIN 索引，下面这条 SQL 在万级数据下也能秒级响应：

```sql
SELECT p.id, p.display_name, p.aliases, p.pinyin
FROM person p
JOIN person_<category>_attrs a ON a.person_id = p.id
WHERE p.status = 'published'
  AND (
        p.display_name ILIKE '%' || :q || '%'
     OR p.pinyin       ILIKE '%' || :q || '%'
     OR EXISTS (SELECT 1 FROM unnest(p.aliases) al WHERE al ILIKE '%' || :q || '%')
  )
ORDER BY similarity(p.display_name, :q) DESC
LIMIT 10;
```

`pg_trgm` 让 `ILIKE '%X%'` 走索引而不是全表扫，**这是 MVP 期不需要外挂 ES 的关键**。

## 3.10 选标签 UI 设计要点

```
[ 1. 选大类（必选 1 个） ]      [ 2. 选标签（同大类内多选，并集） ]
┌─────────────┐                ┌──────────────────────────────────┐
│ ◉ 中国历史  │                │ ☑ 三国    ☐ 秦汉    ☑ 帝王     │
│ ○ 演艺明星  │                │ ☐ 武将    ☑ 文臣    ☐ 外戚     │
│ ○ 体育      │                │ ☐ 谋士                          │
└─────────────┘                └──────────────────────────────────┘
                               题库剩余：247 人  [ 开始 ]
```

- 大类必选 1 个；切换大类时清空标签选择。
- 标签可多选，**取并集**；选 0 个 = 该大类所有人物。
- 实时显示题库剩余人数；< 30 人时弹提示"题库较少，建议放宽"。

## 3.11 与 guessassin 的对照

| 维度 | guessassin | 本设计 |
| --- | --- | --- |
| 题材 | 单一大类（LoL 选手） | 多大类，每局锁定 1 个 |
| 多分类（一人多大类） | 不支持 | **支持（子表存在性表达）** |
| 题库切换 | 中韩 / 全球 | 大类 + 标签并集 |
| 表格列 | 11 列固定 | 每大类独立的列固定 |
| 字段强类型 | 是 | 是（子表 + 类型） |

## 3.14 与现有 `person_memory` 工程的关系

工作区里已有的 `person_memory`、`memory_build_ceping` 等代码可以作为**长期数据源**（人物档案、属性抽取等结果可批量导入），但**项目本身保持独立**，不复用其 ORM/服务，避免相互拖累。
