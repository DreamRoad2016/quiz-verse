# 2. 产品规划与需求

## 2.1 产品定位

| 维度 | 定位 |
| --- | --- |
| 产品形态 | **微信小程序**（首选 Taro/原生 + 云开发或自建后端） |
| 核心玩法 | 类 Wordle / LoLdle 范式的**猜人物**游戏 |
| 内容覆盖 | 中外**王侯将相、才子佳人、明星草根、科技体育**等多类别人物 |
| 模式 | 单人**默认题库**、单人**自选标签**、**多人对战抢答** |
| 受众 | 历史/八卦/体育/科技任一爱好者，泛娱乐用户 |
| 营收 | 暂不商业化；预留：广告位、皮肤、赛季通行证、UGC 题库分成 |

## 2.2 用户故事（User Stories）

### 2.2.1 首次体验（"小白快速上手"）

> 作为一个第一次打开小程序的用户，我希望**3 秒看到一局游戏**，凭直觉就能上手猜，并明白颜色含义。

- 首页直接进入"今日一题"（默认题库随机或当日），无需任何配置。
- 第一次猜测后弹一个**轻提示**说明：绿=完全一致，黄=部分一致，灰=不一致，↑/↓=数值方向。

### 2.2.2 进阶玩家（"我要选自己感兴趣的标签"）

> 作为一个三国迷，我希望**只用三国人物**当题库，能用自定义标签开局。

- 标签系统采用 **"大类约束 + 小标签并集"**（[ADR-001](./DECISIONS.md#adr-001题库组织采用大类约束--小标签并集)）：
  - 玩家先选 **1 个大类**（中国历史 / 演艺明星 / 体育 / 科技 …）
  - 再在该大类内多选 **0~N 个小标签**（如三国、唐、帝王、谋士…）
  - 题库取**并集**：任一标签命中即入选（**不**取交集——交集容易选完一个都没）
  - **不允许跨大类混选**——属性差异太大，无法在同一表格中可比
- 选中后实时显示"题库剩余 N 人"，< 30 人时弹提示建议放宽。

### 2.2.3 朋友对战（"我们仨开个房比比谁先猜中"）

> 作为一群朋友，我希望开一个房间，**大家一起猜同一个目标**，最先猜中者胜。

- 1v1：创建对局 → 分享 uuid 链接（或 6 位邀请码）→ 第二人 join 即开始
- 多人：创建房间 → 分享邀请码 → 房主选标签 → 满 2~8 人后房主点 start
- 实时同步：客户端每 2~3s 拉一次房间快照（[ADR-006](./DECISIONS.md#adr-006对战采用-rest-短轮询--redis-状态mvp-期不引入-websocket)：REST 短轮询 + Redis），看到对手猜测次数与是否命中
- 结算榜：第一名 / 用时 / 猜测次数 / 历史战绩
- **对战必须有独立服务端做权威裁判**（[ADR-003](./DECISIONS.md#adr-003必须独立部署后端服务纯小程序云开发不能承担长期形态)）

### 2.2.4 重度用户（"我想刷天梯"）

> 作为一个重度用户，我希望有**段位排名**和**每日成就**保持长期粘性。

- 排位赛：每日积分制，标签固定（同类玩家同组）。
- 称号 / 段位（青铜→大师），赛季制。

## 2.3 玩法详细规划

### 2.3.1 经典模式（V1 必做）

| 项 | 说明 |
| --- | --- |
| 目标 | 服务端在题库中随机选一名人物为目标（[ADR-002](./DECISIONS.md#adr-002所有比对走服务端不下发完整题库)：客户端不下发答案，比对全在后端） |
| 输入 | 文本框 + 自动联想，输入名/别名/拼音（联想 brief 仅含 id+名+别名+拼音，不含 attrs） |
| 反馈 | 表格逐属性彩色标识，**列 schema 由所选大类决定** |
| 次数限制 | 默认 8 次，可在设置里 6/7/8/9/10 |
| 结束 | 命中或耗尽次数 |
| 题库选择 | 必选 1 个大类 + 大类内多选标签（并集） |

### 2.3.2 时间挑战（V1.5 / V2）

- 限时 60 / 90 / 120 秒，无次数上限，看你能猜中几个目标（连猜模式）。
- 每命中一个 +时间奖励。

### 2.3.3 对战模式（V2 重点）

详见 [`05_对战模式设计.md`](./05_对战模式设计.md)。要点：

- 房间码 / 分享卡片邀请。
- 房主选大类 + 标签 + 次数 + 节奏。
- 服务端唯一裁判（[ADR-002](./DECISIONS.md#adr-002所有比对走服务端不下发完整题库) / [ADR-003](./DECISIONS.md#adr-003必须独立部署后端服务纯小程序云开发不能承担长期形态)）。
- 实时通过 WebSocket 同步每个玩家进度。

### 2.3.4 创意工坊 / UGC（V3）

- 用户提交人物条目，审核后入库。
- 用户/达人创建"自选题包"分享给朋友玩。

## 2.4 内容策略

| 阶段 | 大类范围 | 题库范围 | 数量目标 |
| --- | --- | --- | --- |
| 内测 | `history` + `entertainment` 两个大类 | 三国/秦汉/唐 + 中国当代明星 | 200~300 |
| 公测 | + `sports` + `sci_tech` | + NBA/足球巨星 + 科技大佬 | 800~1500 |
| 长期 | 全 8 个大类 | + 二十四史 + 全球各类垂直领域 + UGC | 5000+ |

> 内容是核心壁垒，**初期手工 + 半自动**整理 200 条精品比一开始凑 5000 条质量低的好。
> 大类不要一次全开——每开 1 个大类都要单独的属性 schema、比对模板、UI 微调。

## 2.5 关键非功能需求（NFR）

| 维度 | 要求 |
| --- | --- |
| 性能 | 单次猜测端到端 < 200ms（含一次后端往返）；对战快照轮询 < 50ms / 间隔 2~3s |
| 包体 | 主包 ≤ 2MB（小程序限制），**题库永远不打包**，按需走 API |
| 兼容 | 微信小程序 iOS / Android 双端；后续考虑 H5 镜像 |
| 容量 | 联想 brief 单大类 < 200KB；attrs 永不下发 |
| 安全 | 所有模式答案均不下发，比对均在服务端（[ADR-002](./DECISIONS.md#adr-002所有比对走服务端不下发完整题库)） |
| 实时 | 对战采用 REST 短轮询而非 WebSocket（[ADR-006](./DECISIONS.md#adr-006对战采用-rest-短轮询--redis-状态mvp-期不引入-websocket)）；"对手命中"感知延迟 ≤ 3s 可接受 |
| 隐私 | 收集 openid / 昵称头像（用户授权）；不收集额外个信 |

## 2.6 版本路线（Roadmap）

| 版本 | 核心交付 | 预计工期（参考） |
| --- | --- | --- |
| **V0.1 内测** | 经典模式单人 + 默认题库（≈200 人） | 3~4 周 |
| **V0.2** | 标签筛选 + 自选题库 + 历史战绩 | +2 周 |
| **V1.0 公测** | 对战房间（2 人） + 排行榜 + 日榜 | +3 周 |
| **V1.5** | 时间挑战 + 多人房间（4~8 人） | +2 周 |
| **V2.0** | 排位赛 + 段位 + 赛季 | +4 周 |
| **V3.0** | UGC 创意工坊 + 内容审核工作流 | +6 周 |

> 上述工期是**1~2 名研发**、产品/设计已就位的乐观估计；个人 side project 可乘 1.5x~2x。

## 2.7 与 guessassin 的差异化

| 维度 | guessassin | 本产品 |
| --- | --- | --- |
| 题材 | 单一大类（LoL 选手） | **多大类**（历史、娱乐、体育、科技、UGC），但每局锁定 1 个大类 |
| 题库切换 | 中韩 / 全球 | **大类 + 标签并集**（[ADR-001](./DECISIONS.md#adr-001题库组织采用大类约束--小标签并集)） |
| 平台 | Web | **微信小程序**（社交分享天然优势） |
| 玩法 | 单机经典 + 时间陷阱 | + **对战房间**、排位、UGC |
| 内容来源 | 维基/官方资料抓取 | 半自动导入 + 编辑校对 + UGC |
| 比对位置 | 客户端 | **服务端**（[ADR-002](./DECISIONS.md#adr-002所有比对走服务端不下发完整题库)） |
| 反作弊 | 弱（前端可见答案） | 强（题库不下发，所有比对在后端） |
| 多分类 | 不支持 | **同一人可归多个大类**（[ADR-004](./DECISIONS.md#adr-004同一人物可归属多个大类多分类)） |
| 对战 | 无 | 1v1 / 多人，REST 短轮询 + Redis（[ADR-006](./DECISIONS.md#adr-006对战采用-rest-短轮询--redis-状态mvp-期不引入-websocket)） |

## 2.8 评估结论

- **方向可行**：标杆产品已验证玩法吸引力，且小程序在中国社交分享场景天然适合"开局猜人物"。
- **核心难点不在玩法**，在 ① 大类 schema 设计与内容录入工作流，② 对战状态管理与反作弊，③ 题库内容长期运营。
- **建议**：先开 `history` + `entertainment` 两个大类，每个大类内 1~2 个标签（如三国、唐 / 当代华语明星）做 MVP 跑通；后续每开新大类视为一次"小立项"，独立排期。
- **本地优先开发**（[ADR-005](./DECISIONS.md#adr-005mvp-后端栈选-javaspring-boot--postgresql--redis本地优先开发)）：Java + 本地 PostgreSQL/Redis + 浏览器 HTML 跑通整套玩法，再上云，最后改造小程序前端。整个过程不依赖云服务也能玩。
- **对战不上 WebSocket**（[ADR-006](./DECISIONS.md#adr-006对战采用-rest-短轮询--redis-状态mvp-期不引入-websocket)）：REST 短轮询 + Redis 即可承载 MVP 期所有对战需求，工程量大幅减少。
