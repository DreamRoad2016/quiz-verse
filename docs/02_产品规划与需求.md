# 2. 产品规划与需求

> **现状（2026-07）**：本地 Web + 题包引擎已可玩（LoL Demo、甄嬛传）。小程序 / 对战 / PG 仍是后续目标，下文「用户故事」保留方向，实现以仓库与 [07 里程碑](./07_里程碑与风险.md) 为准。

## 2.1 产品定位

| 维度 | 定位 |
| --- | --- |
| 当前形态 | **本地 / Web 验证页** + 独立 Spring Boot 后端 |
| 目标形态 | **微信小程序**（接口稳定后换皮） |
| 核心玩法 | 类 Wordle / LoLdle 的**猜人物** |
| 内容组织 | **题包（pack）**；远期大类 + 标签（ADR-001） |
| 模式 | 单人（已有）→ 对战（规划） |
| 受众 | 影视 / 历史 / 体育 / 电竞等爱好者 |
| 营收 | 暂不商业化 |

## 2.2 用户故事（User Stories）

### 2.2.1 首次体验（"小白快速上手"）

> 作为一个第一次打开的用户，我希望**很快进入一局**，凭直觉上手，并明白颜色含义。

- 首页选题包 → 开始新局（当前实现）
- 猜测后表格展示：绿=一致，黄=部分/接近，灰=不一致，↑/↓=数值方向

### 2.2.2 进阶玩家（"我要选自己感兴趣的范围"）

> 作为某个题材的爱好者，我希望只玩该题材。

- **当前**：选择题包（如甄嬛传）即锁定范围
- **远期**：大类 + 小标签并集（[ADR-001](./DECISIONS.md)），选中后提示题库人数

### 2.2.3 朋友对战（规划）

> 作为一群朋友，希望同题竞技。

- 见 [ADR-006](./DECISIONS.md) 与 [05](./05_对战模式设计.md)；**尚未实现**
- 结算榜：第一名 / 用时 / 猜测次数 / 历史战绩
- **对战必须有独立服务端做权威裁判**（[ADR-003](./DECISIONS.md#adr-003必须独立部署后端服务纯小程序云开发不能承担长期形态)）

### 2.2.4 重度用户（"我想刷天梯"）

> 作为一个重度用户，我希望有**段位排名**和**每日成就**保持长期粘性。

- 排位赛：每日积分制，标签固定（同类玩家同组）。
- 称号 / 段位（青铜→大师），赛季制。

## 2.3 玩法详细规划

### 2.3.1 经典模式（当前已做）

| 项 | 说明 |
| --- | --- |
| 目标 | 服务端在题包中随机选一人（ADR-002） |
| 输入 | 联想 brief（id + name + aliases） |
| 反馈 | 表格按题包 schema 着色 |
| 次数 | `pack.yaml` 的 maxGuesses（默认 8） |
| 题库 | 选择题包（远期：大类 + 标签） |

### 2.3.2 时间挑战（V1.5 / V2）

- 限时 60 / 90 / 120 秒，无次数上限，看你能猜中几个目标（连猜模式）。
- 每命中一个 +时间奖励。

### 2.3.3 对战模式（规划）

详见 [`05_对战模式设计.md`](./05_对战模式设计.md)。要点：

- 房间码 / 分享邀请
- 服务端唯一裁判（ADR-002 / ADR-003）
- **REST 短轮询 + Redis**（ADR-006），不上 WebSocket（除非后续升级）

### 2.3.4 创意工坊 / UGC（更远期）

- 用户提交人物或自选题包，审核后发布

## 2.4 内容策略

| 阶段 | 范围 | 数量目标 |
| --- | --- | --- |
| 当前 | LoL Demo + 甄嬛传正式包 | 甄嬛传持续扩角色 |
| 下一阶段 | 再 1～2 个正式题包 | 每包数十～数百精品 |
| 长期 | 多大类 / 多作品 + UGC | 视质量扩量 |

> 内容是壁垒：**精品题包**优于一次性堆量。每开一个新题材都要单独 schema。

## 2.5 关键非功能需求（NFR）

| 维度 | 要求 |
| --- | --- |
| 性能 | 单次猜测端到端 < 200ms（含一次后端往返）；对战快照轮询 < 50ms / 间隔 2~3s |
| 包体 | 主包 ≤ 2MB（小程序限制），**题库永远不打包**，按需走 API |
| 兼容 | 微信小程序 iOS / Android 双端；后续考虑 H5 镜像 |
| 容量 | 联想 brief 单大类 < 200KB；attrs 永不下发 |
| 安全 | 所有模式答案均不下发，比对均在服务端（[ADR-002](./DECISIONS.md#adr-002所有比对走服务端不下发完整题库)） |
| 实时 | 对战采用 REST 短轮询而非 WebSocket（[ADR-006](./DECISIONS.md#adr-006对战采用-rest-短轮询--redis-状态mvp-期不引入-websocket)）；"对手命中"感知延迟 ≤ 3s 可接受 |
| 隐私 | 收集 openid / 昵称头像（用户授权）；不收集额外个信 |

## 2.6 版本路线（Roadmap）

| 版本 | 核心交付 | 预计工期（参考） |
| --- | --- | --- |
| **V0.1 内测** | 经典模式单人 + 默认题库（≈200 人） | 3~4 周 |
| **V0.2** | 标签筛选 + 自选题库 + 历史战绩 | +2 周 |
| **V1.0 公测** | 对战房间（2 人） + 排行榜 + 日榜 | +3 周 |
| **V1.5** | 时间挑战 + 多人房间（4~8 人） | +2 周 |
| **V2.0** | 排位赛 + 段位 + 赛季 | +4 周 |
| **V3.0** | UGC 创意工坊 + 内容审核工作流 | +6 周 |

> 上述工期是**1~2 名研发**、产品/设计已就位的乐观估计；个人 side project 可乘 1.5x~2x。

## 2.7 与 guessassin 的差异化

| 维度 | guessassin | 本产品 |
| --- | --- | --- |
| 题材 | 单一大类（LoL 选手） | **多大类**（历史、娱乐、体育、科技、UGC），但每局锁定 1 个大类 |
| 题库切换 | 中韩 / 全球 | **大类 + 标签并集**（[ADR-001](./DECISIONS.md#adr-001题库组织采用大类约束--小标签并集)） |
| 平台 | Web | **微信小程序**（社交分享天然优势） |
| 玩法 | 单机经典 + 时间陷阱 | + **对战房间**、排位、UGC |
| 内容来源 | 维基/官方资料抓取 | 半自动导入 + 编辑校对 + UGC |
| 比对位置 | 客户端 | **服务端**（[ADR-002](./DECISIONS.md#adr-002所有比对走服务端不下发完整题库)） |
| 反作弊 | 弱（前端可见答案） | 强（题库不下发，所有比对在后端） |
| 多分类 | 不支持 | **同一人可归多个大类**（[ADR-004](./DECISIONS.md#adr-004同一人物可归属多个大类多分类)） |
| 对战 | 无 | 1v1 / 多人，REST 短轮询 + Redis（[ADR-006](./DECISIONS.md#adr-006对战采用-rest-短轮询--redis-状态mvp-期不引入-websocket)） |

## 2.8 评估结论

- **方向可行**：标杆产品已验证玩法吸引力，且小程序在中国社交分享场景天然适合"开局猜人物"。
- **核心难点不在玩法**，在 ① 大类 schema 设计与内容录入工作流，② 对战状态管理与反作弊，③ 题库内容长期运营。
- **建议**：先开 `history` + `entertainment` 两个大类，每个大类内 1~2 个标签（如三国、唐 / 当代华语明星）做 MVP 跑通；后续每开新大类视为一次"小立项"，独立排期。
- **本地优先开发**（[ADR-005](./DECISIONS.md#adr-005mvp-后端栈选-javaspring-boot--postgresql--redis本地优先开发)）：Java + 本地 PostgreSQL/Redis + 浏览器 HTML 跑通整套玩法，再上云，最后改造小程序前端。整个过程不依赖云服务也能玩。
- **对战不上 WebSocket**（[ADR-006](./DECISIONS.md#adr-006对战采用-rest-短轮询--redis-状态mvp-期不引入-websocket)）：REST 短轮询 + Redis 即可承载 MVP 期所有对战需求，工程量大幅减少。
