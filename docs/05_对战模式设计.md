# 5. 对战模式设计

> 本节实现遵循 [ADR-006](./DECISIONS.md#adr-006对战采用-rest-短轮询--redis-状态mvp-期不引入-websocket)：**REST 短轮询 + Redis 状态**，MVP 阶段不引入 WebSocket。
>
> 关键认知：对战的本质是"多个客户端共享同一份对局状态"。状态存在 Redis，所有客户端定时拉取/写入即可——不需要服务端推送也不需要长连接。

## 5.1 玩法概述

> 多名玩家进同一房间，房主**选 1 个大类 + 多选小标签**（题库取并集，[ADR-001](./DECISIONS.md#adr-001题库组织采用大类约束--小标签并集)）开启一局；服务端对所有人统一选定一个答案人物。每人在自己的设备上独立猜测，**最先猜中者**或**用次数/用时少者**获胜。

## 5.2 模式区分：1v1 vs 多人

| 模式 | `match.mode` | 触发开局 | 房主 | 默认胜负规则 |
| --- | --- | --- | --- | --- |
| **1v1 单挑** | `1v1` | 第二人 join 即自动开始 | 无（创建者 = 玩家 1） | `first_hit`（先命中者胜） |
| **多人房间** | `multi` | 房主点击 start（≥2 人）或满员 | 创建者 | `first_hit` / `least_guesses` / `least_time` 三选一 |

> 1v1 极简——分享个 uuid 链接给朋友点开就玩；多人模式才有"房主、准备、开始"等概念。

## 5.3 房间生命周期

```
┌────────┐  POST /matches/{1v1|multi}    ┌──────────┐  第二人 join (1v1) /        ┌──────────┐  hit / 全员耗尽   ┌──────────┐
│ Lobby  │ ─────────────────────────────►│  Waiting │ ─── 房主 start (multi) ───► │  Playing │ ────────────────► │  Settled │
└────────┘                               └──────────┘                             └──────────┘                   └──────────┘
                                                │                                       │                              │
                                              kick / leave                          give up / leave                结算 → 异步入库
                                                                                                                       │
                                                                                                                  Redis TTL 自然清理
```

## 5.4 Redis 数据结构（对局活跃期）

```
match:{uuid}                          Hash    对局元数据
  - answerId            string        答案 person.id（敏感，不外发）
  - categoryId          string
  - tagIds              json string   题库筛选标签
  - mode                string        '1v1' / 'multi'
  - winRule             string        'first_hit' / 'least_guesses' / 'least_time'
  - maxGuesses          int           默认 8
  - status              string        'waiting' / 'playing' / 'settled'
  - hostUserId          string        多人模式房主
  - winnerUserId        string        胜者（settled 后才有）
  - createdAt           ms
  - startedAt           ms
  - endedAt             ms

match:{uuid}:players                  Set     已加入玩家的 userId 列表（最大 8）
match:{uuid}:player:{userId}          Hash    单玩家进度
  - nickname            string
  - joinedAt            ms
  - guessCount          int           已猜次数
  - hit                 0/1
  - hitAt               ms
  - durationMs          int           hit 时长（hitAt - startedAt）
  - status              string        'active' / 'finished'  /  'gave_up'
match:{uuid}:player:{userId}:guesses  List    该玩家全部猜测明细（最新追加在尾）
                                              每条 = JSON({ index, personId, ts, results })

match:{uuid}:lock                     SetNX   原子锁，用于"首杀写入"
match:{uuid}:code                     String  6 位 邀请码 → uuid（多人模式用）
                                              反查：code:{6位码} → uuid
TTL：
  Waiting   → 15 min
  Playing   → 30 min
  Settled   → 5 min（够异步入库 + 让玩家看结算页）
```

> 选用 `Hash + Set + List` 而非全部塞进 JSON：单条字段更新（如 `guessCount += 1`）走 `HINCRBY` 原子操作，不需要 read-modify-write 锁。

## 5.5 接口协议（全 REST，无 WS）

### 5.5.1 创建对局

```http
POST /matches
{
  "mode": "1v1",                       // '1v1' / 'multi'
  "categoryId": "history",
  "tagIds": ["three_kingdoms"],
  "winRule": "first_hit",              // multi 模式可选
  "maxGuesses": 8
}
→ 200
{
  "matchId": "m_abc",                  // 即 uuid
  "code": "X9K2QR",                    // 多人模式才有，1v1 不需要
  "categoryId": "history",
  "schema": { ... },                   // 该大类的比对模板
  "briefList": [ {id, name, aliases, pinyin}, ... ]
}
```

### 5.5.2 加入对局

```http
POST /matches/{uuid}/join              # 1v1 / multi 都用这个
# 或：POST /matches/by-code/{code}/join   # 多人模式凭 6 位码加入
→ 200
{
  "matchId": "m_abc",
  "playerId": "u_123",
  "snapshot": { ... }                  // 当前对局状态
}
```

### 5.5.3 提交一次猜测

```http
POST /matches/{uuid}/guess
{ "guessPersonId": "p_017" }
→ 200
{
  "guessIndex": 3,
  "isHit": false,
  "remaining": 5,
  "results": { "history.dynasty": { "kind": "partial", "matched": ["三国-蜀汉"], "color": "yellow" }, ... },
  "endStatus": null,                   // 自己耗尽次数才会非空
  "matchEnded": false                  // 整局是否已结束（被对手抢先 hit）
}
```

服务端处理流：

```python
def submit_guess(uuid, userId, guessPersonId):
    if redis.hget(f"match:{uuid}", "status") != "playing":
        raise BadState
    answerId = redis.hget(f"match:{uuid}", "answerId")
    categoryId = redis.hget(f"match:{uuid}", "categoryId")

    answer = load_attrs(answerId, categoryId)         # JOIN 主表 + 子表
    guess  = load_attrs(guessPersonId, categoryId)
    schema = SCHEMAS[categoryId]
    results = compare(guess, answer, schema)
    isHit = (guessPersonId == answerId)

    redis.hincrby(f"match:{uuid}:player:{userId}", "guessCount", 1)
    redis.rpush(f"match:{uuid}:player:{userId}:guesses", json.dumps({
        "index": newCount, "personId": guessPersonId, "ts": now_ms(), "results": results
    }))

    if isHit:
        # 首杀原子锁
        if redis.set(f"match:{uuid}:lock", userId, nx=True, ex=60):
            # 我抢到了胜者
            redis.hset(f"match:{uuid}", mapping={
                "status": "settled", "winnerUserId": userId, "endedAt": now_ms()
            })
            redis.expire(f"match:{uuid}", 300)         # 5 min 等持久化
            schedule_async_persist(uuid)               # 入 PostgreSQL
        else:
            # 别人已经先一步抢到，自己 finished
            redis.hset(f"match:{uuid}:player:{userId}", "status", "finished")

    elif newCount >= maxGuesses:
        redis.hset(f"match:{uuid}:player:{userId}", "status", "finished")
        check_all_finished(uuid)                       # 全员结束则 settle

    return build_guess_response(...)
```

### 5.5.4 拉取对局快照（轮询）

```http
GET /matches/{uuid}
→ 200
{
  "matchId": "m_abc",
  "status": "playing",
  "mode": "multi",
  "winRule": "first_hit",
  "categoryId": "history",
  "maxGuesses": 8,
  "startedAt": 1714374000000,
  "players": [
    { "userId": "u_123", "nickname": "Alice",
      "guessCount": 4, "hit": false, "status": "active" },
    { "userId": "u_456", "nickname": "Bob",
      "guessCount": 6, "hit": true, "durationMs": 47000, "status": "finished" }
  ],
  "winnerUserId": null,
  "myGuesses": [ ... ]                  // 仅自己的猜测明细（含 results）
}
```

只返回**他人猜测的统计摘要**（次数、是否命中、耗时），**不下发他人的具体猜了什么人**——保护差异化体验。
房主可以选 `visibility: 'full'`，那时摘要里再带"对方猜过的 personId 列表"。

### 5.5.5 房主 / 离开 / 投降

```http
POST /matches/{uuid}/start             # 多人模式房主点开始
POST /matches/{uuid}/leave             # 离开
POST /matches/{uuid}/give-up           # 投降
```

## 5.6 客户端轮询策略

| 场景 | 频率 |
| --- | --- |
| Waiting（等其他人加入/房主开始） | 3s |
| Playing（活跃猜测中） | 2s |
| 自己已 finished（等对方/结算） | 5s |
| 切后台 / 锁屏 | **暂停**轮询 |
| 切回前台 | 立即拉一次，恢复正常频率 |

* 轮询间隔可由服务端在响应里下发 `nextPollMs` 字段，便于动态调整
* 单次猜测后立即拉一次快照（看自己最新进度 + 别人变化），不必等下一轮

## 5.7 完成入库（异步持久化）

```
触发：match.status 变为 'settled' （首杀 / 全员耗尽）
异步任务：
  1. 写 match 行（mode, categoryId, tagIds, answerPersonId, winnerUserId, startedAt, endedAt, ...）
  2. 写 guess 行（每人每次猜测一行：matchId, userId, guessIndex, guessPersonId, results, recvTime）
  3. 写 user_stat（胜场/总场数等增量）
  4. 写 season_score（积分增量）
  5. 全部成功后才删除 Redis 中的 match:{uuid} 与所有子键
失败重试：
  - 单步失败 retry 3 次（指数退避）
  - 仍失败放入 dead-letter 队列，人工/定时器重试
  - 在 Redis TTL 内（5 min）数据安全
```

> 策略：**对战进行中只用 Redis，结束后才入 PostgreSQL**。避免每次猜测都写 DB，性能最优。

## 5.8 反作弊

| 攻击向量 | 对策 |
| --- | --- |
| 客户端拿到答案 | 任何模式都不下发 `answerId`，仅下发"联想 brief"（id+名+别名+拼音，不含 attrs） |
| 提交不在题库内的 personId | 服务端校验 `guessPersonId` 必须 ∈ briefList |
| 抢答时刻调整本地时间 | 服务端记录 `serverReceiveTimeMs`；客户端时间不参考 |
| 多账号刷胜场 | 同设备 / 同 IP 限制；排位赛绑定微信 openid |
| 偷看 Redis | Redis 在内网，不对公网暴露；接口层 ACL（需要房间成员身份才能 GET 快照） |
| 第三方脚本 | 输入 + 提交都需 session token + nonce + 时间戳签名；防 replay |

## 5.9 容量预估

| 指标 | 估算 |
| --- | --- |
| 单房间人数 | V1.0：≤2（仅 1v1）；V1.1：≤8（multi） |
| 单局 Redis 占用 | ~5 KB（含全员猜测明细） |
| 1k 并发对局 | ~5 MB Redis 内存（无压力） |
| 后端 QPS（轮询） | 1k 房间 × 4 人 × 0.5 QPS ≈ **2k QPS**，单台 Spring Boot 4C8G 轻松扛 |
| MVP 单台机器估容量 | **5k 并发对局** 没问题 |

## 5.10 容错与边界

- **玩家退出**：客户端不再轮询；服务端依靠 TTL 清理；不阻塞其他玩家结算。
- **服务端崩溃重启**：Redis 是状态权威；后端无状态可滚动重启。
- **Redis 主从切换**：可能丢极少数对局（< 1s 写入），可接受。
- **对局超时（30 min 无活动）**：直接 `settled` + winner=null 入库，避免内存泄漏。

## 5.11 后续升级路径（V2+）

- **WebSocket 推送**：当用户量上来 / 实时性要求变高，把"对手猜中事件"通过 WS 推给同房间其他人；REST 接口保留作兜底。
- **战队/天梯**：在 `season_score` 与 `user_stat` 之上加段位算法。
- **观战**：观众订阅房间快照（GET 接口加 `?asObserver=1`），服务端只返回"全员摘要 + 答案揭晓"事件流。
