# 4. 单局猜测算法

> 本节定义猜测算法的输入 / 比对规则 / 输出格式。
> - 所有比对在服务端完成（[ADR-002](./DECISIONS.md#adr-002所有比对走服务端不下发完整题库)）。
> - 比对时按对局的 `categoryId` 从**对应的大类专属子表**读 attrs（[ADR-004](./DECISIONS.md#adr-004同一人物可归属多个大类多分类)：同一人物在不同大类局里展示不同视角属性）。

## 4.1 总体流程

```
[客户端]                                              [服务端]

1. 选大类 + 选标签 → POST /matches             ─►  2. 校验 categoryId、tagIds 都属于该 category
                                                   3. JOIN 对应大类子表筛题库（详见 03 §3.8）
                                                   4. 随机选答案 person.id（不下发）
                                                   5. 返回 { matchId, categoryId, schema, briefList }
                                                       ├─ schema：该大类的比对模板
                                                       └─ briefList：题库内全部 person 的
                                                          { id, displayName, aliases, pinyin }
                                                          —— 仅供联想，不含 attrs

6. 玩家输入并选中下拉 → POST /matches/:id/guess ─► 7. 校验 guessPersonId 必须在题库 brief 内
                                                   8. 按 match.categoryId 读 person_<cat>_attrs：
                                                       guessAttrs = answer 的子表行 + person 主表
                                                       answerAttrs = guess 的子表行 + person 主表
                                                   9. 按 schema 逐列比对
                                                  10. 返回 GuessResult（绿/黄/灰、↑/↓）
                                                  11. 命中或耗尽：返回 endStatus + answerReveal

12. 客户端渲染表格
13. 直至游戏结束
```

要点：
- **比对全部在服务端**，单人/对战逻辑统一，避免客户端拿到答案。
- **按 match.categoryId 切换 attrs 子表**：姚明在 sports 局展示中锋/火箭，在 entertainment 局展示嘉宾/参演节目。
- **客户端只缓存"联想 brief"**（`{id, name, aliases, pinyin}`），不含 attrs，无法反推答案。
- briefList 体积小（千条数量级 < 200KB），首次创建对局时一次性下发即可。

## 4.3 比对类型规则

下方 `G` = 玩家猜测人物的字段值；`A` = 答案人物的字段值。

### 4.3.1 `name`（仅用于 ID/姓名 列）

```
if A.id == G.id  → exact (绿)，并触发 isHit = true
else             → none (灰)
```

### 4.3.2 `single`（单值字符串 / 枚举 / bool）

```
if A is null or G is null → unknown (灰)
elif A == G               → exact (绿)
else                      → none (灰)
```

### 4.3.3 `set`（多值集合）

```
let GS = set(G or []), AS = set(A or [])
if not GS or not AS  → unknown (灰)
elif GS == AS        → exact (绿)
else:
    inter = GS ∩ AS
    if inter ≠ ∅     → partial (黄), matched = list(inter)
    else             → none (灰)
```

> **UI 注意**：`partial` 时把命中的元素 chip 单独高亮（深色块）+ 未命中的浅色，对应 guessassin "卢锡安" 的处理。

### 4.3.4 `number`（数值）

```
if A is null or G is null → unknown (灰)
elif A == G               → exact (绿)
else:
    diff = A - G
    threshold = column.threshold (默认 0)
    arrow = '↑' if diff > 0 else '↓'
    if abs(diff) <= threshold → near (黄), arrow=arrow
    else                       → diff > 0 ? higher (灰↑) : lower (灰↓)
```

各列默认阈值：

| 列 | threshold | 备注 |
| --- | --- | --- |
| `era_century` | 1 | 同/相邻世纪算接近 |
| `birth_year` | 5 |  |
| `debut_year` | 2 |  |
| `awards_count` / `world_events` | 1 |  |
| `championships` | 0 | 严格相等才绿 |

## 4.4 输出类型

```ts
type AttrCompareResult =
  | { kind: 'exact';   color: 'green' }
  | { kind: 'partial'; color: 'yellow'; matched: string[] }
  | { kind: 'none';    color: 'gray' }
  | { kind: 'higher';  color: 'gray';   arrow: '↑' }
  | { kind: 'lower';   color: 'gray';   arrow: '↓' }
  | { kind: 'near';    color: 'yellow'; arrow: '↑' | '↓' }
  | { kind: 'unknown'; color: 'gray' };
```

## 4.5 比对引擎伪代码

```python
def compare(guess: dict, answer: dict, schema: list[Column]) -> dict:
    out = {}
    for col in schema:
        g = read(guess, col.key)
        a = read(answer, col.key)
        ctype = col.compare

        if ctype == 'name':
            out[col.key] = exact() if guess['id'] == answer['id'] else none_()

        elif ctype == 'single':
            if g is None or a is None:
                out[col.key] = unknown()
            elif g == a:
                out[col.key] = exact()
            else:
                out[col.key] = none_()

        elif ctype == 'set':
            gs, as_ = set(g or []), set(a or [])
            if not gs or not as_:
                out[col.key] = unknown()
            elif gs == as_:
                out[col.key] = exact()
            else:
                inter = gs & as_
                out[col.key] = partial(sorted(inter)) if inter else none_()

        elif ctype == 'number':
            if g is None or a is None:
                out[col.key] = unknown()
            elif g == a:
                out[col.key] = exact()
            else:
                diff = a - g
                arrow = '↑' if diff > 0 else '↓'
                threshold = col.threshold or 0
                if abs(diff) <= threshold:
                    out[col.key] = near(arrow)
                else:
                    out[col.key] = (higher if diff > 0 else lower)(arrow)
        else:
            raise ValueError(f"unknown compare type: {ctype}")
    return out
```

## 4.6 服务端职责清单

| 职责 | 说明 |
| --- | --- |
| 校验入参 | `categoryId` 合法、`tagIds` 都属于该 category、`maxGuesses` 在范围内 |
| 题库选取 | 按 [03 §3.8](./03_数据模型设计.md#38-题库筛选规则postgresql) 的 SQL 筛 `person.id`（必 JOIN 对应大类子表） |
| 答案选定 | 在筛选后集合内随机；记录 `match.answer_person_id`（不返回客户端） |
| 联想 brief 下发 | 仅 `{id, name, aliases, pinyin}`，**绝不带 attrs** |
| 单次猜测校验 | `guessPersonId` 必须在题库 brief id 列表内（防绕过） |
| 加载 attrs | **按 `match.categoryId` 读对应子表** + 主表合并；同一人在不同大类局里读不同子表 |
| 比对计算 | 按该大类的 schema 逐列比对 |
| 结束判定 | `isHit` 或 `guessIndex >= maxGuesses` 即结束；记录战绩 |
| 揭晓答案 | 只在 endStatus 非空时返回 `answerReveal: PersonBrief` |

## 4.7 关键性能预算

| 项 | 目标 |
| --- | --- |
| `POST /matches` 端到端 | < 300ms（一次 SQL + briefList 序列化） |
| `POST /matches/:id/guess` 端到端 | < 200ms |
| 联想 briefList 大小 | < 200KB（5000 人级别） |
| 比对引擎单次执行 | < 1ms（10 列以内的纯计算） |

## 4.8 缓存与扩展

- 题库 brief 按 `(category_id, tagSet)` 做 5min 缓存，避免每次创建对局都查 DB。
- answer 选定后将 `answer + guess` 的 attrs 存入 Redis（TTL = 对局时长 + 5min），避免对战每次猜测都查 DB。
- 比对引擎纯函数无状态，水平扩展无压力。

## 4.9 单测/回归测试要点

- 每种比对类型的边界用例（空值、全空、全等、空交集、阈值边界）。
- 模板字段误配（key 不存在、子表 join 失败）应抛 5xx 而非 200 返回乱码。
- 给每个大类至少 5 条样例的"猜 A 对 B"快照（snapshot）测试。
