# 7. 里程碑与风险评估

> 路线总纲（[ADR-005](./DECISIONS.md#adr-005mvp-后端栈选-javaspring-boot--postgresql--redis本地优先开发)）：
>
> **本地 Java + 本地 PostgreSQL/Redis + 本地 HTML/JS 测试页 → 完整跑通 → 上云 → 改造小程序前端**。

## 7.1 里程碑

> 时间以"一个人投入 50% 时间"的 side-project 节奏估，全职团队可压缩到 1/2~1/3。

### M0 — 立项与素材（1 周，全部本地）

- 选定首批 **2 个大类**：`history`（首批：三国）、`entertainment`（首批：当代华语明星 / 选秀）
- 为这 2 个大类各完成属性 schema 与比对模板（[03 数据模型](./03_数据模型设计.md)）
- 完成属性词典（朝代、势力、出道节目名、所属公司枚举…）的初稿
- 跑通 LLM 辅助抽取流程（输入维基条目 → 输出 attrs JSON）
- 本地装好 Docker、Java 17、PostgreSQL 15、Redis 7（建议 `docker-compose.yml` 一把起来）

### M1 — 单人玩法本地跑通（3~4 周，全部本地）

后端（Spring Boot）：
- `person` 主表 + 2 张大类子表 + `tag` / `person_tag` / `category`
- 比对引擎（按 schema + categoryId 计算绿/黄/灰、↑/↓）
- 接口：
  - `POST /matches`（创建单人对局）
  - `POST /matches/{uuid}/guess`（提交一次猜测）
  - `GET /people/search?categoryId&q`（联想搜索，走 `pg_trgm` GIN 索引）
- 题库 ≥ 200 条精选（每大类 ≥ 100 条），手抓 + LLM 辅助
- 本地 Docker PostgreSQL（开启 `pg_trgm` 扩展） + Redis；本地 IDEA 跑后端

前端（本地 HTML/JS）：
- 选大类 → 选标签 → 开局
- 输入联想（仅 brief）→ 提交 → 表格逐列着色
- 命中或耗尽显示结果

✅ **M1 验收**：浏览器能玩一局，三国和明星各跑通

### M2 — 自选标签 + 战绩 + 录入页（2 周，本地）

- 标签 UI 完整（多选、并集、剩余人数提示）
- 用户战绩页（历史对局、命中率、最佳成绩）—— 用 mock userId
- **录入页**（`/admin/people`）：管理后台，方便手工 + LLM 半自动录人物
  - 选大类（多选）→ 显示对应大类的属性表单
  - 提交 → 写主表 + 对应大类子表 + person_tag
- 题库扩到 300+ 条
- 数据反馈通道（玩家点错漏 → 后台工单）

✅ **M2 验收**：录入工作流顺畅，3 天能录 50 条新人物

### M3 — 对战实现（本地）+ 启动备案（3 周）

后端：
- Redis 数据结构（[05 §5.4](./05_对战模式设计.md#54-redis-数据结构对局活跃期)）
- 1v1 接口：`POST /matches/1v1`、`POST /matches/{uuid}/join`、`POST /matches/{uuid}/guess`、`GET /matches/{uuid}`
- 多人接口：增加 `code` 邀请码、`POST /matches/multi`、`POST /matches/{uuid}/start`
- 首杀原子锁（Redis SETNX）
- 异步入库（结束才写 PostgreSQL）

前端：
- 对战房间页（创建 / 加入 / 轮询快照 / 提交猜测 / 结算页）
- 浏览器开两个窗口模拟两个玩家本地对战

并行启动：
- 申请阿里云 / 腾讯云账号
- 申请域名 + 启动 ICP 备案（1~2 周）

✅ **M3 验收**：本地 1v1 + 4 人 multi 都跑通，备案在路上

### M4 — 上云（2 周）

- ECS（4C8G） + RDS for PostgreSQL + 云 Redis
- 域名 + HTTPS（备案下来后接入）
- CI/CD：GitHub Actions → 容器化 → Push 镜像 → 拉到 ECS
- 数据迁移脚本：本地 `pg_dump` → 云 RDS `pg_restore`
- 联调云端环境，把本地 HTML 里的 baseUrl 改成 `https://api.xxx.com`

✅ **M4 验收**：浏览器访问公网域名能完整玩，性能达标

### M5 — 改造小程序前端（3~4 周）

- HTML 改写为 Taro 小程序工程（保留接口契约）
- 微信登录（`wx.login` → 后端 code2session）替换 mock userId
- 房间分享卡片
- 第一次提审 / 公测

✅ **M5 验收**：小程序内测版上架，朋友能扫码玩

### M6 — 多人 / 排行榜 / 时间挑战（3 周）

- 多人对战上限到 8 人
- 时间挑战模式（90s 连猜）
- 日榜 / 周榜 / 总榜
- 每日成就

### M7 — 排位赛 / 段位 / 赛季（4 周）

- 段位算法（青铜→大师）、赛季制
- 排位匹配（按段位分组）

### M8 — UGC 创意工坊（6 周，远期）

- 用户提交人物 / 题包
- 审核流（机审 + 人审）
- 创作者激励

## 7.2 风险与应对

| 风险 | 影响 | 应对 |
| --- | --- | --- |
| **题库内容是壁垒，量大就累人** | 高 | MVP 用"窄而精"策略；大类逐个开；中长期借助 LLM + 社区 UGC |
| **跨题材属性差异大，统一表难处理** | 高 | **大类约束 + 子表分离**（[ADR-001](./DECISIONS.md#adr-001题库组织采用大类约束--小标签并集) / [ADR-004](./DECISIONS.md#adr-004同一人物可归属多个大类多分类)）已在 03 设计 |
| **多分类增加录入工作量** | 中 | 录入分阶段：MVP 期姚明先录 sports，后期补 ent 即可；模型支持但运营按需补全 |
| **本地 → 云迁移有坑（数据/网络/HTTPS）** | 中 | M3 开始**并行启动备案**；M4 预留 1~2 周迁云调试时间 |
| **轮询延迟带来"对战实时性"体验差** | 低-中 | 自适应频率（活跃 2s）；后续 V2 升级 WS（[ADR-006 折衷](./DECISIONS.md#adr-006对战采用-rest-短轮询--redis-状态mvp-期不引入-websocket)）可平滑替换 |
| **Redis 单点故障** | 中 | 上云后用主从/Cluster；MVP 期单实例 + 入库容错（结束才入库 + 重试机制） |
| **PostgreSQL 大版本升级** | 低 | 用 `pg_dump`/`pg_restore` 迁移；用稳定的 15.x，5 年内不需要升级 |
| **小程序包体限制（主包 2MB）** | 中 | 题库永不打包，按需走 API；联想 brief 单大类 < 200KB |
| **客户端作弊（拿到答案）** | 低 | [ADR-002](./DECISIONS.md#adr-002所有比对走服务端不下发完整题库)：所有比对在后端，答案永不下发 |
| **冷启动用户少，对战凑不齐人** | 中 | 主推 1v1（双人即可玩） + 朋友局；适时加 AI Bot |
| **内容合规（历史/政治/名人姓名争议）** | 中 | 人物白名单制；运营后台一键下架；优先选公认中性人物 |
| **数据来源版权** | 低-中 | 维基/百科允许引用，标注来源；不直接复制大段文字 |
| **DAU 增长依赖社交分享** | 中 | 设计天然分享点：每日一题战绩卡、对战邀请卡、回放卡 |
| **成本失控** | 低 | MVP 全本地；上云后 < 500 元/月；LLM 用本地模型/批量调用 |

## 7.3 是否值得做？决策建议

> 综合评估：**值得做**，节奏建议如下：

### 7.3.1 立项前可做（不需要正式立项就能干）

1. **本地 `docker-compose` 起 PostgreSQL + Redis**（10 分钟），并开启 `pg_trgm` 扩展
2. **本地 Spring Boot 起手项目**（半天搞定，工程脚手架；JDBC 连 PG）
3. **录 2 个大类各 5~10 条样例 JSON**（建 `samples/history/`、`samples/entertainment/`）
4. **写 100 行的 `ComparisonEngine.java`**：读取两条人物 JSON + schema → 输出比对结果，命令行 demo
5. **画 2~3 张 UI 草稿**（首页、对局表格、对战房间），找朋友看反馈

### 7.3.2 立项门槛（满足以下任一可正式立项）

- 至少 200 条高质量人物数据**在 2 个大类内**跑通端到端
- 找到 1~2 个共建者（至少有一个对前端/UI 不抵触）
- 自己能持续每周投入 ≥ 10 小时

### 7.3.3 不建议的方向

- **一次性做全题材**：会被属性差异拖死，2 个大类跑通再扩
- **死磕排位/付费**：先把"开局猜得过瘾"做好
- **盲目大而全 UGC**：审核成本极高，等核心用户证明留存后再做
- **MVP 阶段就上 WebSocket**：用 REST 短轮询能拖到几千 DAU，没必要提早上 WS
