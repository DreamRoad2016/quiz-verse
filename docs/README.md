## 核心决策一览（详见 `DECISIONS.md`）

| ID | 决策 | 一句话 |
| --- | --- | --- |
| **ADR-001** | 题库组织 | **大类约束 + 小标签并集**：每局必选 1 个大类，大类内多选标签取并集；不允许跨大类混题 |
| **ADR-002** | 比对位置 | **所有比对走服务端**，客户端只缓存"联想 brief"（id+名+别名+拼音），attrs 永不下发 |
| **ADR-003** | 后端形态 | **必须有独立后端**；不能纯小程序云开发兜底 |
| **ADR-004** | 多分类 | **同一人物可归多个大类**：通过"在该大类的子表里有记录"自然表达，姚明能同时在 sports + entertainment 题库出现 |
| **ADR-005** | 技术栈 | **Java + Spring Boot + PostgreSQL 15 + Redis，本地优先**；MVP 用浏览器 HTML 验证，后改造小程序 |
| **ADR-006** | 对战实时性 | **REST 短轮询 + Redis 状态**；MVP 不上 WebSocket，工程量大幅减少；V2 再视体验升级 |

## 一句话总结

> 做一款类 LoLdle/Wordle 范式的微信小程序：
> - 题库覆盖**多大类**（历史 / 演艺 / 体育 / 科技 …），但每局锁 1 个大类 + 大类内标签并集
> - **同一人物可归多个大类**（姚明可在体育局也可在演艺局当答案）
> - **所有比对在独立后端**完成，单人/对战逻辑统一
> - **本地 Java + PostgreSQL + Redis 全栈跑通**再上云；**对战用 REST 短轮询**，MVP 不上 WS

## 标杆游戏一句话

`guessassin.xyz` 是猜 LoL 职业选手的网页游戏，每局选定一个目标选手，玩家最多 8 次猜测，每次提交后表格逐列以 **绿/黄/灰** 表示属性匹配程度，数值列还会用 **↑/↓** 提示目标值方向。所有选手数据通过一次 `GET /api/players/{regionPool}` 拉到前端，**比对完全在客户端完成**（这个做法我们**不沿用**——见 [ADR-002](./DECISIONS.md#adr-002所有比对走服务端不下发完整题库)）。
