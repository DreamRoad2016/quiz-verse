> 本节遵循：
> - [ADR-003](./DECISIONS.md#adr-003必须独立部署后端服务纯小程序云开发不能承担长期形态)：必须有独立后端

## 6.0 为什么必须有独立后端？

> 详见 [ADR-003](./DECISIONS.md#adr-003必须独立部署后端服务纯小程序云开发不能承担长期形态)。

### 6.0.1 微信小程序的能力边界

| 维度 | 限制 |
| --- | --- |
| 运行环境 | 沙箱化 JS，不能开端口、不能监听连接 |
| 长连接 | 仅能作为客户端连出去 |
| 后台运行 | 切后台/锁屏可能被冻结，**无法持续承载房间状态** |
| 跨用户共享 | 小程序之间无法直接通信，必须经由共同的服务端 |
| 数据存储 | 本地 `Storage` 仅限本机；多人对局必须中心化 |
| 业务安全 | 客户端可被反编译，敏感比对逻辑不能放前端 |

### 6.0.2 三件事必须由"独立服务"做

1. **题库存储与按需下发**（不下发答案 → 减体积、防扒库）
2. **比对裁判**（[ADR-002](./DECISIONS.md#adr-002所有比对走服务端不下发完整题库)）
3. **多人对局状态共享**（在 Redis 中以 uuid 为键）

> 结论：**必须有一个独立可控的后端**。

## 6.1 总体架构（MVP 版，本地优先）

```
┌────────────────────────────────────┐                ┌────────────────────────────────────────────┐
│  本地浏览器 (HTML + JS)              │                │       本地 Spring Boot 应用 (Java 17)        │
│   - guessOne 单人页                 │  HTTP/JSON     │  ┌────────────────────────────────────────┐ │
│   - matchRoom 对战页（轮询拉快照）   │ ─────────────► │  │ Controller                              │ │
│   - admin 录入页（运营录入人物）     │                │  │   /matches /guess /people /admin/...    │ │
│                                     │                │  ├────────────────────────────────────────┤ │
└────────────────────────────────────┘                │  │ Service                                 │ │
                                                       │  │   MatchService / ComparisonEngine /     │ │
                                                       │  │   PersonService / TagService / Auth     │ │
                                                       │  ├────────────────────────────────────────┤ │
                                                       │  │ Persistence                             │ │
                                                       │  │   MyBatis-Plus / Spring Data JPA        │ │
                                                       │  └─────────┬──────────────────────┬────────┘ │
                                                       └────────────┼──────────────────────┼──────────┘
                                                                    │                      │
                                                       ┌────────────▼──────┐    ┌──────────▼─────────┐
                                                       │  本地 PostgreSQL  │    │  本地 Redis 7      │
                                                       │  - person 主表    │    │  - 活跃对局状态    │
                                                       │  - 大类专属子表   │    │  - 邀请码反查      │
```

> **没有 WebSocket Hub**——对战靠 REST 短轮询读 Redis，详见 [05_对战模式设计](./05_对战模式设计.md#54-redis-数据结构对局活跃期)。

## 6.2 阶段化部署形态

```
M0~M2：纯本地（IDEA + 本地 Docker PostgreSQL/Redis + 浏览器 HTML）
        ↓ 单人玩法 + 数据 + 接口契约稳定
M3：本地实现对战（Redis 状态 + 短轮询）
        ↓ 跑通 1v1 + multi
M4：上云（迁阿里云/腾讯云）
        - ECS（4C8G） + RDS for PostgreSQL + 云 Redis
        - 域名备案、HTTPS（备案 1-2 周，M3 阶段并行启动）
M5：改造小程序前端
        - HTML + JS 改写为 Taro/原生小程序
        - 接口契约几乎不动；只是从 fetch 换成 wx.request
        - 之前的 HTML 测试页保留作"接口验证用例库"
```

## 6.3 前端

### 6.3.1 MVP 阶段：本地 HTML + JS

| 项 | 选型 | 备注 |
| --- | --- | --- |
| 入口 | 静态 `index.html` | 直接浏览器打开或 Spring Boot 静态资源托管 |
| 框架 | Vanilla JS / Vue 3（CDN 引入） / Alpine.js | 不上构建工具，简单为先 |
| HTTP | `fetch` API | 轮询用 `setInterval` |
| 样式 | Tailwind via CDN 或纯 CSS | 不追求美观，能看就行 |
| 域名 | `localhost:8080` 或 `127.0.0.1:8080` | 后端同源，无 CORS 问题 |

> 这一层目标只有一个：**验证后端接口契约 + 跑通玩法**。

### 6.3.2 正式阶段：微信小程序

| 项 | 选型 | 理由 |
| --- | --- | --- |
| 框架 | **Taro 4 + React + TypeScript** 或微信原生 + TS | 一套代码 → 小程序 + H5；TS 类型安全 |
| 状态 | Zustand / Pinia | 轻量 |
| UI | NutUI / Vant Weapp | 组件齐全 |
| 网络 | 封装 `wx.request`（替换 MVP 阶段 fetch） | 接口契约不变 |
| 缓存 | `wx.setStorage` | briefList 等 |

## 6.4 后端

| 项 | 选型 | 理由 |
| --- | --- | --- |
| 语言 | **Java 17** | 团队最熟 |
| 框架 | **Spring Boot 3** | Java 后端事实标准；起手快 |
| 运行 | Tomcat 嵌入式 | Spring Boot 默认 |
| ORM | **MyBatis-Plus**（推荐）或 Spring Data JPA | 多表 JOIN 多，MyBatis 灵活；两者对 PG 完全友好 |
| 数据库 | **PostgreSQL 15**（本地起；上云用 RDS for PG） | 多值集合 `text[]` + GIN、`pg_trgm` 中文模糊匹配、`jsonb` 都是原生强项；详见 [ADR-005](./DECISIONS.md#adr-005mvp-后端栈选-javaspring-boot--postgresql--redis本地优先开发) |
| 数据库驱动 | `org.postgresql:postgresql` JDBC 42.x | 官方 JDBC，性能稳定 |
| PG 必备扩展 | `pg_trgm`（联想搜索）、`uuid-ossp` 或内置 `gen_random_uuid()` | 都是 contrib 模块，本地 Docker 镜像默认带 |
| 缓存 | **Redis 7**（本地 Docker；上云用云 Redis） | 对局状态 + 题库 brief 缓存 |
| 序列化 | Jackson | Spring Boot 默认 |
| 接口文档 | springdoc-openapi（Swagger UI） | 写完接口直接看文档 |
| 微信登录 | 上云后 `wx.login` → code2session 接口 | MVP 期可先 mock userId |

### 6.4.1 关键服务模块

| 模块 | 职责 |
| --- | --- |
| `auth` | 微信登录、JWT、用户档案（MVP 期可先 mock） |
| `content` | 人物 CRUD、大类、标签、子表录入 |
| `match` | 单人 / 对战对局生命周期、答案选定、Redis 状态、入库 |
| `comparison` | 比对引擎（按 schema + categoryId 计算绿/黄/灰、↑/↓） |
| `stats` | 战绩、排行榜（V2 起） |
| `admin` | 内容审核、UGC 提交（V2 起） |

### 6.4.2 关键性能预算

| 指标 | 目标 |
| --- | --- |
| `POST /matches` | < 300ms（一次 SQL + briefList 序列化） |
| `POST /matches/{uuid}/guess` | < 200ms（Redis 读 + 比对 + Redis 写） |
| `GET /matches/{uuid}` 快照（轮询） | < 50ms（纯 Redis 读） |
| `GET /people/search?q=`（联想搜索） | < 30ms（GIN trigram 索引） |
| 联想 brief 单大类下发 | < 200KB（gzip 后） |
| MVP 单台机器并发对局 | 5k 对局 / 2k QPS（4C8G 轻松扛） |

## 6.5 数据库（PostgreSQL 15）

详见 [03_数据模型设计](./03_数据模型设计.md)。要点：

```
person                          共性主表（无 category_id，参 ADR-004）
category                        大类定义
tag                             标签（依附大类）
person_tag                      人物-标签 多对多
person_history_attrs            历史人物视角
person_arts_attrs               文人/艺术家视角
person_entertainment_attrs      演艺/明星视角
person_sports_attrs             体育视角
person_sci_tech_attrs           科技/学者视角
person_business_attrs           商业视角
person_grassroots_attrs         网红视角
person_mythology_attrs          神话视角

user                            用户
match                           对局历史（仅持久化已结束的，活跃中在 Redis）
guess                           猜测明细（同上）
user_stat / season / season_score   战绩与排位（V2 起）
```

> 多值集合用 **`text[]`** + GIN 索引；中文联想搜索用 **`pg_trgm`** 扩展（不需要外挂 ES）。

## 6.6 Redis（活跃对局 + 题库缓存）

详见 [05 §5.4](./05_对战模式设计.md#54-redis-数据结构对局活跃期)。

```
match:{uuid}                       Hash 对局元数据
match:{uuid}:players               Set
match:{uuid}:player:{userId}       Hash
match:{uuid}:player:{userId}:guesses  List
match:{uuid}:lock                  SetNX 首杀原子锁

```

## 6.9 监控与运营

- **业务埋点**：进入率、完成率、命中率、平均次数、对战胜率、分享率
- **质量埋点**：API 失败率、平均响应时间、Redis 命中率、PG 慢查询（`pg_stat_statements`）
- **报警**：5xx 比例、Redis 内存、PG CPU、连接数
- **后台**：内容审核 / 玩家举报 / 题库版本管理 / 配置开关

## 6.10 安全与合规

- 微信小程序备案 + 内容合规审核（人物涉及历史/政治敏感时下架）
- 用户内容（UGC）走机器审核 + 人工复核
- 个人信息收集最小化（仅 openid + 授权头像昵称）
- 接口防刷：rate limit、签名、nonce 防 replay
